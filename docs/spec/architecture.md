# Архітектура

## Обраний стиль
Проєкт використовує багатошарову архітектуру:
- `controllers` приймають HTTP-запити
- `services` містять бізнес-логіку
- `repositories` ізолюють роботу з MongoDB
- `models` описують доменні сутності
- `dto` тримають контракти API

## Чому MongoDB
- оголошення, повідомлення та галереї природно лягають у документну модель;
- швидше еволюціонувати схему без SQL-міграцій;
- зручний варіант для marketplace-подібного каталогу з фільтрацією;
- PyMongo дає простий контроль над індексами та запитами.

## Патерни
- `Repository` для ізоляції persistence-логіки
- `Service Layer` для бізнес-правил

## SOLID
- `SRP`: контролери, сервіси й репозиторії мають різні відповідальності
- `OCP`: нові сценарії додаються через нові сервіси/репозиторії без ламання API
- `LSP`: сервіси працюють із доменними об’єктами, а не з конкретним драйвером
- `ISP`: залежності вузькі, без "універсального god-object"
- `DIP`: верхні шари залежать від абстракції repository/session, а не від raw Mongo calls
