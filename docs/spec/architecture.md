# Архітектура Та Патерни

## Архітектурний стиль
Проєкт використовує багатошарову архітектуру:
- `controllers`: HTTP-рівень, маршрути, статуси, залежності.
- `services`: бізнес-правила та оркестрація сценаріїв.
- `repositories`: доступ до даних через шаблон Repository.
- `models`: ORM-сутності.
- `dto`: контракти запитів і відповідей.

Такий підхід добре підходить для навчального проєкту, бо робить код прозорим для тестування, рефакторингу та розширення.

## GoF / архітектурні шаблони
- `Repository`: ізолює SQLAlchemy та запити від бізнес-логіки.
- `Singleton`: конфігурація `Settings` кешується через `@lru_cache`, щоб мати єдине джерело налаштувань.

## Структура шарів
- `controllers`: CRUD-ендпоїнти, маршрути автентифікації, модерації, користувачів і повідомлень.
- `services`: правила модерації, перевірки прав доступу, обробка бізнес-сценаріїв.
- `repositories`: повторно використовуваний доступ до БД.
- `core`: безпека, конфігурація, життєвий цикл застосунку.

## SOLID
- `S`: контролери не містять бізнес-логіку; сервіси відповідають за сценарії; репозиторії за доступ до БД.
- `O`: нові сценарії можна додавати через нові сервіси й DTO без переписування існуючих маршрутів.
- `L`: базовий `Repository` підмінюється конкретними репозиторіями без зміни контракту.
- `I`: DTO та залежності вузькі й спеціалізовані для окремих сценаріїв.
- `D`: контролери залежать від абстракції сервісного шару та dependency injection FastAPI.

## Код-smells і рефакторинг
- Початковий smell: неповний CRUD. Виправлення: додано окремі контролери та сервіси для `users`, `categories`, `listings`, `messages`.
- Початковий smell: декларативний Docker без робочого PostgreSQL-драйвера. Виправлення: додано `psycopg[binary]`, `.env.example`, healthcheck БД і retry під час ініціалізації схеми.
- Початковий smell: відсутність операційного спостереження. Виправлення: додано middleware логування HTTP-запитів і подій сервісного шару.
